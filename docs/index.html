Which pages need a refresh first? A decision-support model for content prioritization

A ranking model that triples baseline precision at identifying declining content, validated under a client-grouped holdout split, with an honest audit of what it can and cannot claim.

Selva Kumaran G · FlyRank ML Internship, 2026

Abstract

Content teams must decide which of thousands of pages to refresh first, but simple rules based on a single metric perform poorly. Using a 30,000-page anonymized content-performance dataset, I built a Logistic Regression model to rank pages by decline risk, using features like ranking position, staleness, and engagement — explicitly excluding any field the label itself was derived from. Under a client-grouped holdout split (no client's pages seen in both train and test), the model reached a Precision@50 of 0.70, more than double a transparent, non-leaky baseline rule's 0.32. The result is decision-support, not causal proof: it ranks pages worth a human's attention, not a guarantee that refreshing any single page will reverse its decline.

Introduction / Problem statement

Every content team eventually hits the same wall: traffic has plateaued or declined across a large site, and there isn't enough writer or editor time to refresh everything. The real decision is triage — which pages should be reviewed first, this cycle, given limited hours? A wrong call has a real cost either way: refresh a page that wasn't actually failing, and that time is wasted while a genuinely declining page goes untouched; miss a real decline, and the traffic loss compounds unnoticed until the next review.

This paper asks a narrow, answerable question: can a simple, transparent model rank pages by decline risk meaningfully better than a hand-written rule, on the same data and the same honest validation split? The output is meant to sit in front of a human reviewer as a prioritized shortlist — not to run unattended.

Data

This analysis uses the FlyRank ML Internship starter release: content_refresh_anonymized.csv, 30,000 pseudonymized pages across 32 anonymized clients, with 44 columns covering search performance (impressions, clicks, CTR, average position), content attributes (word count, content type, age), and a 90-day trend snapshot.

  • Unit of analysis: one row = one page, at one point-in-time snapshot, for one client.
  • Time window: a single 90-day performance snapshot — not a time-series of repeated visits to the same page.
  • Excluded from features: trend_direction and trend_pct — the label (is_declining_label) is derived directly from trend_direction, so using either as a model input would leak the answer into the question.
  • Public-safe: the dataset ships pre-anonymized (pseudonymized page and client IDs); no client names, domains, or raw queries appear anywhere in this analysis or repository.

Methodology

Label

is_declining_label = (trend_direction == "down"). This is a proxy, not ground truth on intent-to-refresh — a page can decline for reasons a refresh won't fix (seasonality, a lost SERP feature, a stronger competitor), and a stable page might still benefit from a proactive update. Base rate: 54.2% of pages in this sample are labeled declining.

Baseline (non-leaky, by design)

An early draft of this baseline directly gated on trend_direction and scored a suspicious Precision@50 of 1.0 — a clear leakage signal, since the rule could not lose against a metric built from the same field it used as an input. The corrected baseline instead blends two independent, pre-decline signals: a visibility percentile (log-scaled 90-day impressions) and a freshness-risk percentile (days since last update), weighted equally. This is the baseline the model is compared against below.

Features

13 numeric features (search volume, competition, CPC, word/char count, days with impressions, days since last update, CTR, average position, engagement rate, scroll rate, AI-traffic share, content age) and 3 categorical features (competition level, content type, main search intent) — all observed at the same snapshot as the label, none derived from it.

Validation design

Split by client, not row (GroupShuffleSplit, 80/20, seed 42) — no client's pages appear in both train and test. This matters: a naive random row split let 31 of 32 clients leak across both sides, which inflated Precision@50 to an overly optimistic 0.78. The honest, client-grouped number is 0.70 — an ~8-point gap that was pure leakage, not real generalization.

Leakage audit

Final feature set re-checked against the label source: trend_direction and trend_pct confirmed absent. client_id used only for the split, never as a model input. No feature showed a correlation with the label above ~0.16 in magnitude — consistent with real, moderate signal rather than a disguised copy of the label.

Notably, the simpler Logistic Regression outperformed Random Forest on this split — a reminder that added model complexity isn't automatically added performance, and readable models deserve a fair first look.

An honest negative result: staleness alone is a weak signal

A natural hypothesis is that staler content declines more. The bucketed data doesn't cleanly support this: decline rate was 51.2% for pages updated in the last 90 days, rose to 61.1% for 91–180 days, then fell to 46.7% for 181–365 days (n=169) and 60.0% for 365+ days (n=5, too small to trust). Staleness on its own is mixed, not confirmed — it only earns a place in the model alongside other signals, not as a standalone rule.

Limitations & honest framing

  • This is a single 90-day snapshot of one anonymized portfolio (32 clients) — not validated across other time periods, industries, or content types.
  • The label is observed and directional, not causal: the model ranks pages that look similar to past declining pages. It does not identify why a page is declining, and it does not predict or reverse-engineer any search engine's ranking algorithm.
  • Precision@50 ≈ 0.70 means roughly 3 in 10 flagged pages are still false positives — this is decision-support for prioritizing human review, not an automated action trigger.
  • The staleness signal, on its own, is mixed in this sample (see above) — a caution against over-trusting any single "obvious" feature without checking it.
  • All numbers here are specific to one seed, one split, and one dataset; they should be read as "observed in this data," not as general claims about content decay.

Human review is mandatory

Before acting on any flagged page, a reviewer should confirm: is the decline seasonal or industry-wide? Did the page recently lose a SERP feature unrelated to content quality? Is the page still strategically relevant? Does the reason code actually match what a human sees on the page?

What should never be automated

  • No auto-publishing or auto-editing based on the score alone.
  • No auto-deprioritizing or hiding pages from search — "no action" means "not flagged this cycle," not "safe to remove."
  • No client-facing guarantee of traffic recovery from any refresh.
  • No skipping human review for the thinnest-evidence rows — those need the most scrutiny, not the least.

Monitoring & retrain triggers

Re-validate if Precision@50 on fresh data drops meaningfully below ~0.70, if feature distributions shift noticeably from this training snapshot, or if the volume of flagged pages swings sharply between runs. At minimum, re-check quarterly.
